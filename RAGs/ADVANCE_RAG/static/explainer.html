1 What is Retrieval-Augmented Generation?

RAG combines information retrieval with LLM generation. Instead of asking a model to answer from its training data alone, we:

🔍
Retrieve
Find relevant documents
→
🔗
Augment
Add context to prompt
→
🤖
Generate
LLM answers with citations

This approach grounds LLM responses in verified data, eliminating hallucinations and enabling question-answering over private corpora.

3 Components Deep Dive

🔍 Hybrid Retrieval — bge-m3 ▼

BAAI/bge-m3 is a multi-vector encoder that produces both:

  • Dense vectors (1024-dim) — semantic understanding
  • Sparse vectors (vocab-size) — exact token matching

This is hybrid retrieval — the best of both worlds. Dense captures meaning ("create campaign" ≈ "set up experiment"), sparse captures precision ("VWO-1234" matches exactly).

BAAI/bge-m3 ~2.3 GB 1024-dim dense
🗄️ Vector Database — Qdrant ▼

Qdrant is a high-performance vector database that natively supports:

  • Dense + sparse vectors in the same collection
  • Payload filtering — filter by priority, module, JIRA ID
  • Embedded mode — runs as a file store, no Docker required
  • Batch search — search both dense and sparse in one call

The collection vwo_test_cases stores each chunk with its dense vector, sparse vector, and metadata payload.

⚡ Reciprocal Rank Fusion (RRF) ▼

RRF combines the rankings from dense and sparse searches into a single ranked list:

RRFscore(d) = Σ₁/(k + rank(d))

Where k = 60 (smoothing constant)

This prevents either method from dominating — a document that ranks highly in both searches gets a much better score than one that ranks highly in only one.

🎯 Cross-Encoder Re-ranking ▼

BAAI/bge-reranker-v2-m3 is a cross-encoder that evaluates query-document pairs jointly (not just vector similarity).

  • Takes the top 20 candidates from hybrid search
  • Scores each pair (query, doc) with full attention
  • Returns the top 4 most relevant chunks for the LLM
  • ~570 MB model, loaded on demand

This is significantly more accurate than bi-encoder similarity, but slower — so we only run it on the top candidates.

BAAI/bge-reranker-v2-m3 Cross-encoder
✏️ Query Rewriting ▼

Before searching, we generate 3 alternative phrasings of the user's question using a fast LLM via Openrouter:

Original: "How do I create an A/B test campaign?"
Rewrites:
  • "steps to set up an A/B test campaign"
  • "A/B campaign creation workflow"
  • "how to launch a split test experiment"

This captures different phrasings and increases recall — the same question asked differently might match different chunks.

🤖 LLM Generation (Openrouter) ▼

The final step uses DeepSeek Chat (via Openrouter) to generate a grounded answer from the retrieved chunks.

Two modes are auto-detected:

  • Answer mode — answers questions with [Chunk N] citations
  • Generate mode — creates structured test cases using retrieved templates

The system prompt enforces grounded answers — the LLM must use only the provided context.

6 The Data — 5,000 VWO Test Cases

The corpus consists of 5,000 realistic test cases for VWO (Visual Website Optimizer), an A/B testing platform. Test cases span 10 modules:

Campaign Management User Segmentation Reporting & Analytics Integrations Account Settings User Management Billing API & Webhooks Audience Targeting Personalization

Each test case includes: JIRA ID, title, steps, expected results, priority (P0-P3), module, tags, and preconditions.

9 Running It Yourself

python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Set your API key in .env
#    OPENROUTER_API_KEY=sk-...

# 4. Run the app
python app.py
# Open http://localhost:5050

# 5. (Optional) In压 test cases via CLI
python ingest.py testcase/vwo_test_cases.csv \
  --text-cols title,steps,expected \
  --meta-cols id,jira_id,priority,module,tags
