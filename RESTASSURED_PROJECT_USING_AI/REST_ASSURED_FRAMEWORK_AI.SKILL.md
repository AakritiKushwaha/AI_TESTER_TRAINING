**Mandatory "Don't" rules:**
1. Do Separate Test Logic from Request Logic
Follow the Page Object Model (POM) or a Layered Architecture variant for APIs. Your actual test classes should only contain assertions and test data flows, not hardcoded endpoints, headers, or query parameters.

API Actions/Endpoints Layer: Handles the construction of requests (e.g., UserService.java, AuthService.java).

Test Layer: Calls these service methods and asserts the results (e.g., UserTests.java).

2. Do Use POJOs for Serialization and Deserialization
Avoid passing massive JSON payloads as raw Strings or complex Maps. Use libraries like Jackson or Gson alongside Rest Assured to map JSON to Plain Old Java Objects (POJOs).

Use tools like Lombok (@Data, @Builder) to reduce boilerplate code when creating payload models.

3. Do Abstract Common Specifications Using RequestSpecBuilder and ResponseSpecBuilder
Instead of repeating base URLs, content types, and authentication details in every single request, bundle them into reusable specifications.

4. Do Implement Robust Logging and Reporting
When an API test fails in a CI/CD pipeline, you need to know exactly what went wrong without rerun-ing it locally.

Use Rest Assured's built-in filters: .log().ifValidationFails() or .log().all().

Integrate reporting tools like Allure Report or Extent Reports to visually capture request/response logs, headers, and status codes upon failure.

5. Do Validate JSON Schemas
Don't just assert a few field values (like id or status). Ensure the structural integrity of the API response by performing JSON Schema Validation. This catches unexpected data type changes or missing fields from the backend immediately.

❌ The Don'ts: Mistakes to Avoid
1. Don't Hardcode Test Data or Environments
Never hardcode base URLs, environment variables, user credentials, or dynamic test data directly inside your test code.

Instead: Use a configuration manager (like config.properties, application.yaml, or environment variables) to switch seamlessly between DEV, STAGE, and PROD.

2. Don't Rely Heavily on Static Thread Sleep
Avoid using Thread.sleep() while waiting for an asynchronous API process to complete. It slows down execution pipelines unnecessarily.

Instead: Use libraries like Awaitility to poll the API dynamically until the desired condition or state is met.

3. Don't Chain Assertions Blindly Without Hard/Soft Strategies
If a test checks 5 different fields in a response, a single failure at line 1 should not always obscure failures down the line unless it's a blocker.

Instead: Use Soft Assertions (via TestNG or AssertJ) when validating multiple independent fields in a JSON response so you get a complete picture of all discrepancies in a single run.

4. Don't Create Interdependent Tests
API tests should be completely stateless and independent. Test B should never rely on the database state or dynamic IDs generated by Test A.

Instead: Use setup hooks (@BeforeMethod / @BeforeClass) to dynamically generate necessary prerequisite data (e.g., creating a user before trying to delete them) and clean up after execution.

5. Don't Ignore Performance: Reuse HTTP Connections
If you create a fresh client instance configuration or repeatedly fetch authentication tokens for every single HTTP request, your test suite will crawl.

Instead: Fetch auth tokens once per suite run (or cache them if valid) and let Rest Assured efficiently leverage the underlying HTTP client connection pooling.
